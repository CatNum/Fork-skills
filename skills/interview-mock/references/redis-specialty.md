# Redis 专项

用户选择 `C. Redis 专项` 后读取本文件。

## 面试目标

考察候选人是否把 Redis 理解为低延迟状态系统：数据结构、内存、持久化、高可用、缓存一致性和故障模式。

## 重点方向

- 数据结构：string、hash、list、set、zset、stream、bitmap、HyperLogLog、GEO、底层编码选择。
- 执行模型：单线程命令执行、I/O 线程、事件循环、慢命令影响。
- 内存：淘汰策略、过期策略、大 key、热 key、内存碎片、内存分配器行为。
- 持久化：RDB、AOF、rewrite（重写）、fsync 策略、崩溃恢复取舍。
- 高可用：主从复制、Sentinel、Cluster、slot 迁移、故障切换、脑裂风险。
- 缓存模式：cache-aside、write-through、write-behind、缓存穿透、缓存击穿、缓存雪崩。
- 一致性：数据库和缓存更新顺序、延迟双删、版本号、订阅 binlog。
- 分布式协调：分布式锁、Redlock 争议、fencing token（栅栏令牌）、幂等、限流。
- 运维：延迟诊断、slowlog、commandstats、keyspace scan、容量规划。

## 提问方式

围绕生产现象提问：突然延迟升高、缓存 miss 风暴、热 key、内存增长、故障切换后不一致、重复处理、锁失效。

避免在问题里暴露预期分类。

## P7 标准

强回答应包含：

- 命令级或机制级推理。
- 有风险意识的止血方案。
- 一致性边界解释。
- 可观测性信号。
- 降级方案。

## 追问策略

- 如果回答说“加锁”，追问如何避免旧 owner 删除新锁。
- 如果回答说“缓存预热”，追问部分失败时会发生什么。
- 如果回答说“Cluster”，追问多 key 操作会有什么变化。
- 如果回答忽略内存，追问如何安全地发现和处理大 key。
