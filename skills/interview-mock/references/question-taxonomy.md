# 面试题型分类

用于在模拟面试和学习资料沉淀时区分 **实战题**、**八股题** 和 **机制追问题**。分类只服务于内部判断和复盘沉淀；正式向用户提问时，问题本身不要泄露答题方向。

## 分类标准

### 实战题

从真实业务、线上故障、系统演进或项目经历切入，要求候选人给出判断、排查、止血、取舍和复盘。

典型特征：

- 问题里有线上现象、业务背景、约束条件或故障表现。
- 需要回答排查步骤、验证证据、临时止血、长期治理。
- 可以追问候选人在项目中的角色、边界判断和复盘改进。

示例：

- 订单列表接口突然从几十毫秒变成 2 到 3 秒，你怎么定位和止血？
- 线上出现死锁告警，你怎么判断是哪两条 SQL 互相等待？
- 主从延迟导致用户刚写入的数据查不到，你怎么保证读己之写？

### 八股题

从概念、定义、原理或机制本身切入，主要考察候选人是否掌握基础知识和标准表达。

典型特征：

- 问题直接问“是什么、为什么、区别是什么、原理是什么”。
- 不要求先处理线上场景，也不强调止血顺序。
- 适合快速查漏补缺，但高难度模拟面试中不能长期停留在纯定义层。

示例：

- 什么是 MVCC？
- redo log 和 binlog 有什么区别？
- B+Tree 为什么适合做数据库索引？

### 机制追问题

由实战题或八股题自然追深出来，聚焦一个容易答错的机制边界。它通常不是完整线上故障题，但也不是宽泛定义题。

典型特征：

- 问题很窄，常围绕“到底会不会、如何判断、边界是什么”。
- 需要把结论、判断标准、反例和优化延伸讲清楚。
- 适合作为独立学习资料沉淀，不要合并回原题。

示例：

- `LIMIT 10000,20` 前面被跳过的 10000 条到底会不会回表？你怎么判断？
- 当前读和快照读在加锁行为上有什么边界？
- 有索引但优化器不选这个索引，可能是什么原因？

## 出题策略

- 所有专项都必须先尊重用户选择的题型模式，再生成第一题。
- **实战题模式**：只问实战题和由实战题自然追出来的机制追问题。除非用户明确要求补基础，否则不要主动切到纯八股。
- **八股题模式**：只问八股题和由八股题自然追出来的机制追问题。不要把问题包装成线上故障、项目事故或开放系统设计。
- **混合模式**：默认优先问实战题，尤其是 MySQL、Redis、系统设计、工程化和项目专项；如果用户基础明显薄弱，或连续在机制上答错，可以插入八股题做补基础；如果用户在实战题中暴露了关键机制漏洞，优先追机制追问题。
- 不要在问题文本里写“这是实战题/八股题/机制追问题”，除非用户明确要求查看分类。
- 不要把优化方向、答题框架或关键词写进问题；这些只出现在点评、标准答案或学习资料里。

## 各专项适配

- Go、MySQL、Redis：三种题型都适用。实战题关注线上现象和排查；八股题关注语言/数据库/中间件机制；机制追问题关注边界判断。
- 项目、工程化、Agent、系统设计、JD 整体面试：默认更适合实战题或混合模式；如果用户选择八股题模式，也要把题目收敛成概念、机制、原则、区别和设计理论，不要展开成完整项目事故。

## 学习资料沉淀规则

沉淀每道题时，不要新增单独的 `## 题型分类` 字段。题型分类必须直接写在问题小标题上。

```markdown
## 1. 【实战题】订单列表分页查询突然变慢，如何定位和止血？

## 2. 【八股题】什么是 MVCC？

## 3. 【机制追问题（由1追问得出）】`LIMIT 10000,20` 前面被跳过的 10000 条到底会不会回表？你怎么判断？
```

如果一道题从原题追问出来，也要作为新的独立问答追加到文件末尾，并在标题中标明来源，例如 `【机制追问题（由1追问得出）】`。如果无法确定来源题号，只写 `【机制追问题】`。
