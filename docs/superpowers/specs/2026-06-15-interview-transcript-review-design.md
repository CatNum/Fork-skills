# Interview Transcript Review Skill Design

## 重点

- **Skill 定位**：面向真实模拟面试视频转写稿，结合简历、岗位 JD 和面试基础信息，生成可复习的面试后复盘文档。
- **核心流程**：先判断原稿是否可读；不可读时允许直接在原文件上格式化；复盘始终创建新的复盘文件。
- **复盘目标**：不是完整会议纪要，而是用于复习的记忆唤醒、薄弱点补齐和下一轮训练。
- **输出分层**：答得还可以的问题只做精炼要点；答得不好的问题补充精炼答案和详细面试版答案；短板用卡片结构扩展。
- **输出格式**：用户可选择输出 Markdown、HTML，或两者同时生成；HTML 更适合观赏、分享和集中复习。

## 一、目标与边界

### 1.1 目标

创建一个新 skill，建议命名为 `interview-transcript-review`。它用于处理真实模拟面试视频转写后的 Markdown 文稿，并结合简历、岗位 JD 和面试基础信息，输出一份新的面试后复盘文档。

该 skill 需要帮助 Codex/Claude 完成三件事：

1. 判断原始转写稿是否可读。
2. 提醒用户补充简历、岗位 JD 和面试基础信息，用于建立复盘上下文。
3. 基于可读原稿生成复盘文件。

### 1.2 输入

输入材料分为必需项和建议项：

1. **面试转写稿（必需）**：一个或多个 `.md` 文件，主输入必须是真实模拟面试视频的转写稿。
2. **简历（建议提供）**：用于判断候选人的真实项目经历、技术背景、职责边界和可追溯素材。
3. **岗位 JD（建议提供）**：用于判断岗位关注点、技能要求、级别预期和复盘优先级。
4. **面试基础信息（建议提供）**：用于判断本轮复盘语境，包括：
   - 面试类型：技术面 / HR 面 / 交叉面 / 项目深挖 / 系统设计 / 其他。
   - 面试轮次：技术一面、技术二面、终面、HR 面等。
   - 目标岗位：如后端开发、AI Agent 开发、平台工程等。
   - 面试时间或批次：可选，用于文件命名和复盘归档。

流程开始时应提醒用户提供简历、岗位 JD 和面试基础信息，但这些不是强制门槛。用户没有提供时，仍继续处理转写稿；只是在相关分析中标注限制。例如：没有简历时，项目经验方向只能基于面试转写稿中已出现的信息判断，无法系统核对候选人的完整项目经历。

面试转写稿可能存在以下情况：

- 已有基本问答结构。
- 只有一长串连续文字，缺少段落、说话人、题答边界。
- 说话段落混在一起，阅读时难以区分停顿和转折。
- 识别错字、拼写错误、术语名词错误较多。
- 问题、追问、回答、点评混在一起。

### 1.3 输出

输出包括：

1. 原稿文件：如果原稿不可读，直接在原文件上格式化为人类可读版。
2. 复盘文件：必须新建，不能覆盖原稿；用户可选择输出 `.md`、`.html`，或两者同时生成。

复盘文件用于复习，不追求记录所有原文细节。它应帮助用户快速回忆本次面试表现，并补齐暴露出的短板。

输出格式规则：

- 流程开始时必须询问用户希望输出哪种格式，不设置默认值。
- **仅 Markdown**：适合继续编辑、纳入知识库和版本管理。
- **仅 HTML**：适合更具观赏性的阅读、分享和复习展示；内容结构与 Markdown 保持一致，但可增加更清晰的视觉层级、评分标签、短板卡片样式和重点高亮。
- **Markdown + HTML**：同时输出两份内容一致、形态不同的复盘文件。

### 1.4 不做什么

- 不生成“可复用回答素材”章节。
- 不把所有问题都扩展成长答案。
- 不把原文格式化阶段变成答案润色阶段。
- 不把用户没有做过的项目经历包装成真实经历；但可以补充技术原理、通用设计方案和学习认识。
- 不覆盖原稿生成复盘；复盘必须写入新文件。

## 二、原文格式化设计

### 2.1 判断是否需要格式化

当原稿出现以下任一情况时，应先做可读化清洗：

- 大段连续文字难以阅读。
- 无明显段落或换行。
- 说话段落没有整理清楚。
- 识别错字、拼写错误、术语名词错误影响理解。
- 标点、断句、换行明显不符合自然表达。

若原稿已经易读，则不强制重写原稿，只在复盘中引用其内容。

### 2.2 原文格式化规则

原文格式化只做可读化清洗，不做题目级结构化。

- 将一长串文字拆成自然说话段落。
- 调整标点、换行和断句，让原文能顺畅阅读。
- 修正明显识别错的文字。
- 修正识别错的拼写、英文缩写、技术术语和专有名词。
- 统一同一术语的写法。
- 保留原始问答顺序。
- 必要时删除无意义的重复口头禅和转写噪声，但不能删除会影响判断的回答内容。

格式化禁止做结构化复盘或语义美化：

- 不把候选人的原回答改写成标准答案。
- 不替候选人补充没有说出的事实。
- 不强制拆成“第 N 题 / 面试官 / 候选人 / 追问”结构。
- 不在原稿中加入评分、短板、补充答案或复盘结论。
- 不删除会影响复盘判断的错误表达、犹豫、遗漏和逻辑断裂。

### 2.3 原稿格式化无模板

原稿格式化之后不使用固定模板，不新增标题层级，不强制标注角色，也不重组为题目清单。只做以下动作：

- 整理自然说话段落。
- 调整标点、换行、断句。
- 修正识别错字。
- 修正拼写、英文缩写、技术术语、专有名词。
- 统一术语写法。
- 保留原始问答顺序和原意。

原稿格式化不能修改内容含义。只能通过换行、断句、标点和必要的错字/术语修正，让每个说话段落更容易阅读。

## 三、复盘文件结构

### 3.1 推荐文件命名

Markdown 复盘文件名建议使用：

```text
<原文件名>-复盘.md
```

HTML 复盘文件名建议使用：

```text
<原文件名>-复盘.html
```

如果用户指定输出路径或文件名，优先使用用户指定。

若用户要求两者同时生成，则同时创建上述 `.md` 和 `.html` 两个文件。

## 四、问题分层规则

### 4.1 回答方向

不要强行把每道题二选一地分成“技术原理类”或“实际项目类”。很多面试问题天然需要混合回答：既可以讲过往项目经验，也可以讲技术原理、设计方法和学习认识。

每道题都必须从两个方向复盘：

| 回答方向 | 使用条件 | 复盘处理方式 |
| --- | --- | --- |
| 项目经验方向 | 用户在过往项目中有类似经验，或面试回答中提到真实项目事实、动作、指标、问题和结果 | 复盘真实项目回答是否讲清背景、个人职责、落地细节、指标结果、故障处理和工程判断 |
| 技术扩展方向 | 用户没有对应项目经历，或项目回答不足以支撑面试官判断，或题目本身更适合考察技术知识广度与深度 | 尽可能扩展和补充知识点，补齐定义、原理、机制、优缺点、适用场景、常见误区、设计方案和追问方向 |

复盘时的原则：

- **项目经验方向必须出现**：如果项目中存在类似经验，就写清真实项目背景、个人动作、结果和边界；如果没有对应经验，就明确写“本次面试未体现对应项目经验”或“过往项目中未直接涉及”，不要硬编项目经历。
- **技术扩展方向必须出现**：无论是否有项目经验，都要从技术知识、原理机制、方案设计、优缺点、适用场景和追问方向上扩展。
- 同一道题必须分清边界：哪些是做过的，哪些是知识扩展或方案推演。
- 不要因为用户没做过某个项目能力，就判断用户不懂这个技术；复盘要评估其知识面、理解深度和表达能力。

### 4.2 回答评分标准

每道题必须给出三级评分，并说明关键依据：

| 档次 | 含义 | 判定标准 |
| --- | --- | --- |
| 通过 | 当前回答基本能支撑面试官继续往下聊 | 回答切题，有清晰主线，关键事实可信，能讲出必要的背景、做法、结果或取舍 |
| 勉强通过 | 方向大体对，但存在明显扣分点 | 有部分有效信息，但结构松散、关键细节不足、指标/取舍/边界不清，容易被追问打穿 |
| 不通过 | 当前回答会显著损害面试评价 | 答非所问、概念错误、项目事实缺失、逻辑断裂、无法说明个人贡献，或暴露严重知识短板 |

评分时优先看 5 个维度：

- **切题性**：是否回答了面试官真正问的问题。
- **结构性**：是否有背景、动作、结果、取舍或边界。
- **可信度**：项目经验方向是否有真实项目细节、指标、角色边界或落地证据；技术扩展方向是否有清晰定义、机制、边界和推理依据。
- **技术深度**：是否讲清原理、难点、方案比较或故障处理。
- **表达稳定性**：是否能让面试官听懂并愿意继续追问，而不是需要猜测。

### 4.3 答得还可以的问题

这类问题评分通常为“通过”或“勉强通过”。它们不需要生成长篇优化答案，复盘目标是唤起用户记忆。

应记录：

- 原题。
- 面试官考点：说明面试官问这道题是为了考察什么能力或风险点，不要只复述题目。
- 回答评分与一句依据。
- 项目经验方向：简述本次回答中体现的项目经验；若没有，明确写未体现。
- 技术扩展方向：用 1 到 3 条要点补充相关技术理解或复习抓手。
- 用户回答中的关键抓手。
- 2 到 5 条精炼要点。
- 一句复习提示。

不应记录：

- 长篇标准答案。
- 与本题无关的泛化知识。
- 可复用回答素材章节。

### 4.4 答得不好的问题

这类问题评分为“不通过”，或虽然是“勉强通过”但暴露了高影响短板。它们需要补答案，分成两层：

1. 简单精炼版答案：按点概括，适合复习时快速扫读。
2. 详细面试版答案：可以直接在面试中说出来的完整表达。

详细面试版答案应遵守：

- 结构清楚，可口头复述。
- 对齐面试官考点，优先补齐题目真正考察的能力维度。
- 补齐原回答缺失的定义、背景、原因、取舍、结果或边界。
- 技术扩展方向要尽可能扩展和补充知识点，体现知识广度与深度。
- 项目经验方向不能把不存在的项目经历说成做过；但可以补充“技术上可以如何设计”“我学习后对这个问题的认识”“如果项目要补齐会怎么做”。
- 若答案同时包含项目经验和技术扩展，必须分清两者边界，不把学习认识伪装成真实经历。

## 五、短板卡片设计

### 5.1 卡片字段

每个短板使用固定字段：

- `面试暴露`：指出来自哪一题、哪段回答或哪类表达问题。
- `问题本质`：判断短板根因，例如知识缺口、表达结构差、缺指标、缺 trade-off、缺故障复盘、可信细节不足。
- `影响等级`：高 / 中 / 低。
- `需要补齐的知识`：列出后续应补的具体知识或材料。
- `下次回答抓手`：给出下一次回答时可直接使用的结构。
- `训练题`：给 1 到 3 道针对性训练题。

### 5.2 短板扩展原则

短板扩展要比逐题复盘更有训练价值。扩展时优先回答：

- 这个短板在面试官视角意味着什么风险？
- 它影响的是能力判断、项目可信度，还是表达清晰度？
- 下次如何用更稳定的回答结构规避？
- 用户需要回到哪些项目材料、知识点或指标口径里补证据？

## 六、执行流程

### 6.1 主流程

1. 读取用户指定的 `.md` 面试转写稿。
2. 询问用户希望输出哪种格式：仅 Markdown、仅 HTML、或 Markdown + HTML。
3. 提醒用户可补充简历、岗位 JD 和面试基础信息，并说明这些信息会提升岗位匹配和项目经验分析质量。
4. 若用户已提供简历、岗位 JD 或面试基础信息，读取并建立复盘上下文；若未提供，继续后续流程，并在复盘中标注对应分析限制。
5. 判断原稿是否可读。
6. 若不可读，先在原稿上做可读化清洗。
7. 从可读化后的原稿中识别问题、回答、追问和反馈。
8. 结合已提供的简历和 JD 判断面试官考点、岗位匹配风险和复盘优先级；缺失材料时只基于转写稿判断。
9. 每道题都从“项目经验方向”和“技术扩展方向”两个维度生成复盘。
10. 按评分标准为每道题标注“通过 / 勉强通过 / 不通过”。
11. 将问题分为“答得还可以”和“答得不好”。
12. 为答得不好的问题补充两版答案，并在技术扩展方向中补足知识深度。
13. 提炼面试暴露出的短板，生成短板卡片。
14. 按用户选择新建复盘 `.md`、`.html`，或同时新建两者。
15. 自检复盘文件是否满足结构和边界要求。

### 6.2 质量自检

- [ ] 原稿不可读时，是否已直接做可读化清洗？
- [ ] 原稿格式化是否只整理段落、断句、错字、拼写和术语，没有美化答案？
- [ ] 原稿格式化是否没有强制拆成题目级结构？
- [ ] 是否已在流程开始时询问用户输出格式，而不是使用默认值？
- [ ] 是否已在流程开始时提醒用户可补充简历、岗位 JD 和面试基础信息？
- [ ] 若缺少简历、岗位 JD 或面试基础信息，是否继续流程并标注对应分析限制？
- [ ] 若用户提供了相关信息，面试概览是否包含目标岗位、JD 匹配重点、面试类型和面试轮次？
- [ ] 复盘是否按用户选择写入新的 `.md`、`.html`，或同时写入两者？
- [ ] 若输出 HTML，内容结构是否与复盘模板一致，并通过视觉层级提升可读性？
- [ ] 每道题是否都包含“项目经验方向”和“技术扩展方向”两个字段？
- [ ] 每道题是否都有“通过 / 勉强通过 / 不通过”三级评分？
- [ ] 每道题是否都有“面试官考点”，且不是简单复述题干？
- [ ] 评分是否基于切题性、结构性、可信度、技术深度、表达稳定性，而不是主观印象？
- [ ] 答得还可以的问题是否只做记忆唤醒式精炼？
- [ ] 答得不好的问题是否包含简单精炼版答案和详细面试版答案？
- [ ] 是否没有生成“可复用回答素材”章节？
- [ ] 短板是否用卡片结构记录并扩展？
- [ ] 项目经验方向是否区分了真实项目经验和技术扩展，没有把学习认识伪装成真实经历？
- [ ] 技术扩展方向是否充分补充了原理、机制、边界、优缺点、适用场景和追问方向？
- [ ] 复盘是否适合后续复习，而不是冗长会议纪要？

## 七、测试思路

### 7.1 测试用例

建议准备 3 类测试 prompt：

1. 无格式长文稿：验证能否先格式化原稿，再新建复盘。
2. 已有问答结构文稿：验证不会无意义重写原稿，只生成复盘。
3. 混合质量回答文稿：验证能否区分答得还可以和答得不好，并为短板生成卡片。

### 7.2 验收标准

生成的 skill 应能让模型稳定做到：

- 识别真实模拟面试转写稿。
- 区分原稿格式化与复盘生成两个阶段。
- 对不同质量的问题采用不同深度处理。
- 用短板卡片沉淀训练材料。
- 保持复盘文档简洁、可复习、可追溯。
